## 0. Executive Summary

이 키트는 **"안전 제일 + 한국 팀 최적화"** 포지션. 참조 모델(Pandey 5-Layer, Anthropic 공식 Agent Skills)의 표준을 정확히 따르면서도, 한국 팀의 경험 — 3-파일 배치 규칙, 🔴 위험 파일 재확인, 사용자 승인 게이트 — 을 **강제 메커니즘**으로 박아 넣음.

외부 툴과 비교했을 때 **절대적 규모에서는 열세**(wshobson 112 agents vs 우리 7, BMAD 50+ workflows vs 우리 7)지만, **로컬 스택 결합도·안전 게이트·한국어 워크플로우·프로젝트 convention 변수화**에서 독보적.

한 줄 평가: **"대규모 생태계 대신, 단일 팀의 안전한 반복을 위해 최적화된 소형 플러그인."**

## 2. 장점 (Strengths)

### ✅ 1. 결정론적 안전 게이트 (Determinism > Prompt Trust)
- `hooks/pre-tool-use.sh` — 🔴 위험 파일 수정 시 **sh-level 차단**. 프롬프트에 "하지 마"라고 쓰는 게 아니라 **실행 레이어에서 deny**.
- `hooks/block-dangerous.sh` — `rm -rf`, `git push --force`, `--no-verify`를 **정규식 매처**로 하드 차단.
- PostToolUse가 `tsc --noEmit`를 파일별로 즉시 실행 → 실패 시 `decision: block` + 에러를 컨텍스트에 주입 → implementer가 **사용자 개입 없이** 재시도.
- **경쟁 비교**: BMAD, wshobson, Cline 모두 프롬프트 기반 가드가 대부분. Hook 레벨 차단은 공식 Anthropic 스킬조차 드물게 사용.

### ✅ 2. 3-파일 배치 규칙의 물리적 강제
- Pandey/BMAD/wshobson 모두 "small atomic tasks"를 **권고**하지만, **카운터로 강제**하는 건 이 키트뿐.
- 세션별 `$TMPDIR/adk-<session>/batch-count`로 유니크 파일 집계 → 4번째 파일 수정 시도 시 `permissionDecision: ask`.
- `SubagentStop` 훅이 implementer 종료 시 자동 리셋.
- **실제 효과**: 에이전트가 "이 김에 저것도 고칠게"로 폭주하는 패턴을 구조적으로 차단.

### ✅ 3. AST 기반 영향 분석 (실증 완료)
- `scripts/impact-analyzer.mjs` — `ts-morph`의 `findReferencesAsNodes()`로 **타입 인식 caller 추적** + `madge`로 **순환 의존성 탐지**.
- 실측: `src/lib/constants.ts` 분석 시 26 exports → 14 unique callers → 🔴 score 6 → exit 2 (정확히 의도대로 동작).
- **grep 기반 경쟁 툴 대비**: 주석·문자열·동명이인 false positive 없음. JSON 출력으로 대시보드 연동 가능.
- 실행 시 감지된 Windows 경로 정규화 버그까지 수정 완료 (ts-morph는 `/`, `path.resolve()`는 `\` 반환 이슈).

### ✅ 4. 4+3 Subagent 파이프라인 — 순차 안전 + 병렬 속도
- **순차 안전 게이트**: designer → checker → implementer (승인 게이트 3개, 각 단계 사용자 개입)
- **병렬 속도**: verifier가 code-reviewer / test-runner / security-scanner를 **단일 메시지로 병렬 dispatch** (Claude Code 공식 패턴 적용)
- 각 에이전트에 **`tools` 화이트리스트** 적용: code-reviewer는 Read/Grep/Glob만, implementer만 Edit/Write 권한 → 권한 최소화 원칙 구현.
- **비교**: wshobson의 112개는 대부분 느슨한 역할 정의. 명시적 권한 제한 + 순차/병렬 혼합은 이 키트의 차별점.

### ✅ 5. 프로젝트 convention 변수화
- `implementer.md`가 `constants_path`, `types_path`, `validations_path`, `api_helper_path`, `pagination_path`, `async_hook_path`를 **CLAUDE.md에서 읽도록** 설계.
- 변수가 없으면 **에이전트가 사용자에게 질문** — 하드코딩된 경로 재사용 금지.
- **비교**: BMAD/wshobson은 대부분 범용 에이전트라 프로젝트 경로 결합이 느슨. 여기는 "이 팀의 convention을 CLAUDE.md에서 선언 → 키트가 그걸 참조" 하는 **2단계 구조** — 같은 키트로 여러 프로젝트 커버 가능.

### ✅ 6. 한국어 워크플로우 우선 + 영문 공존
- 에이전트 보고서 포맷이 **한국어**로 통일 (설계 보고서, 의존성 체크, 검증 결과 등)
- `.md/설계문서/수정위험도.md`, `의존성맵.md`, `INDEX.md` 같은 **한국어 경로**를 키트가 기본 참조
- 영문 테크니컬 섹션(SKILL.md description, 훅 스크립트)은 영문으로 유지 — Claude의 매칭 엔진과 호환.

### ✅ 7. 표준 준수 → 생태계 편입 가능성
- `.claude-plugin/marketplace.json`이 Claude Code 공식 스키마 정확히 준수 → `claude plugin marketplace add`로 즉시 설치됨 (검증 완료).
- git push만 하면 **외부 팀도 동일 한 줄로 설치**. npm/github/url 소스로 배포 확장 여지.
- **대조**: 기존 키트는 `cp` + `install.sh` 방식 — 이 작업으로 생태계 호환으로 전환.

## 3. 단점 (Weaknesses)

### ❌ 1. 규모·다양성 부족
- 에이전트 7개 vs wshobson 112개, BMAD 21개, Anthropic Skills 1367개.
- 프론트엔드·DB·DevOps·마케팅 등 **도메인 특화 에이전트 0개** — marketing-saas 전용이면서도 정작 "마케팅 카피 작성 전문 에이전트" 같은 건 없음.
- 신규 프로젝트에 설치해도 즉시 가치 있는 도메인 에이전트가 부족 → 팀이 스스로 추가해야 함.

### ❌ 2. Windows/Bash 결합
- 훅이 **전부 bash 스크립트** — 원칙상 Windows에서는 Git Bash/WSL 필수.
- Windows PowerShell 네이티브 대응 없음 (`shell: powershell` 옵션 활용 안 함).
- `hooks/pre-tool-use.sh`의 `$TMPDIR` 기본값이 Unix 가정 — Windows에서도 Git Bash 안에서는 동작하지만, PowerShell 직접 실행 시 실패.

### ❌ 3. 테스트·CI 커버리지 0
- 키트 자체에 **테스트가 없음** — impact-analyzer, 훅 스크립트, SKILL.md 파싱 모두 수동 검증에 의존.
- `claude plugin validate` 호출 자동화 없음 → 스키마 에러 조기 감지 불가.
- CHANGELOG, semver 릴리스 플로우 없음 → 사용자가 업그레이드 영향을 예측 못 함.

### ❌ 4. 정량 지표 수집 엔진 부재
- 분석 보고서에서 "정량 지표 부재"를 단점으로 지적했는데, 해결책으로 `--json` 출력만 추가. **실제 대시보드·DB 적재 파이프라인은 미구현**.
- SKILLS 실행 횟수, 배치 차단 횟수, verifier 병렬 시간, PostToolUse 재시도율 등 **운영 지표 수집 0**.
- `SessionStart` 훅으로 로그 초기화 → `Stop` 훅으로 메트릭 플러시 하는 파이프라인 없음.

### ❌ 5. MCP 의존성 은폐
- `plugin.json`에 GitHub/Postgres/claude-context MCP가 인라인 → **env 변수 없으면 자동 설치 시 에러 가능성**.
- 사용자가 `GITHUB_TOKEN`, `DATABASE_URL_READONLY` 같은 걸 **미리 세팅하지 않으면 설치 후 혼란**.
- "필수 vs 선택" 구분이 README에만 있고, plugin.json 자체에는 표시 없음.

### ❌ 6. 에이전트 간 통신 프로토콜 없음
- designer → checker → implementer 간 **공유 컨텍스트는 "대화 기록"** — 구조화된 메시지 교환 없음.
- "설계 보고서가 대화에 있을 것"이라고 가정하지만, 긴 세션에서 auto-compaction 발생 시 날아갈 수 있음.
- 파일 기반 상태(`.md/설계문서/`)에 의존 → 에이전트가 이 파일을 **매번 다시 읽어야 함**.

### ❌ 7. 학습·개선 루프 부재
- Hook이 PostToolUse로 실패를 **주입**은 하지만, **누적 학습**(반복 실패 → 패턴 인식 → 규칙 생성)은 없음.
- 어떤 파일에서 🔴 차단이 자주 발생하는지 추적 안 됨 → 위험도 자동 재조정 불가.
- wshobson·BMAD도 같은 한계지만, 규모가 커서 커뮤니티가 패턴을 축적. 우리는 단일 팀이라 느림.

## 4. 외부 툴 비교

### 4.1 wshobson/agents
- **규모**: 72 plugins, 112 agents, 146 skills, 16 orchestrators
- **철학**: production-ready 오케스트레이션 marketplace, 카테고리 23개
- **장점**: 압도적 다양성, 즉시 사용 가능한 도메인 에이전트 풀
- **단점**: 개별 에이전트의 품질·안전 게이트는 느슨. 프로젝트 결합 없음.
- **우리와의 차이**: 우리는 **수량 1/16, 안전 밀도는 훨씬 높음**. wshobson은 "무기고", 우리는 "체크리스트".
- **혼용 전략**: 기본 파이프라인(designer→verifier)은 우리 키트로, 도메인 전문 실행은 wshobson 에이전트를 **@-mention으로 임포트**. 상호 보완 가능.

### 4.2 BMAD-METHOD
- **규모**: 21 specialized agents, 50+ workflows
- **철학**: Business Methodology for AI Development — agile lifecycle 완전 커버 (PM, architect, dev, QA, Scrum Master, BA 등 Role-based)
- **장점**: ideation → deployment까지 전체 SDLC. PM 페르소나·스토리 생성·스프린트 플래닝 내장.
- **단점**: 무겁다. 소규모 팀에는 과잉. 주로 greenfield 프로젝트에 최적화.
- **우리와의 차이**: BMAD가 **"팀 전체"** 를 AI 에이전트로 시뮬레이션한다면, 우리는 **"신중한 개인 개발자의 체크리스트"**. 다른 문제를 풀고 있음.
- **혼용 전략**: 어렵다. 철학이 다르다.

### 4.3 Cline (Plan/Act 모드)
- **규모**: VS Code 확장 단일 제품
- **철학**: Plan mode(분석)와 Act mode(실행)를 분리 — 우리의 designer/checker vs implementer 분리와 **거의 동일한 철학**
- **장점**: 네이티브 VS Code UI, 브라우저 자동화·웹 테스트 내장
- **단점**: 플러그인 시스템 부재. 커스터마이징은 시스템 프롬프트 편집만 가능.
- **우리와의 차이**: Cline은 "2-mode 단일 에이전트", 우리는 "7-agent 파이프라인". 철학은 같지만 우리가 더 세분화.
- **혼용 전략**: Cline 사용자가 이 키트의 `designer.md`/`checker.md` 텍스트를 시스템 프롬프트에 복붙 가능.

### 4.4 Aider
- **규모**: CLI 단일 도구, 모델 선택 자유
- **철학**: Git-first, commit-per-turn, 최소한의 추상화
- **장점**: 52.7% 벤치마크 점수, 257초 / 126k 토큰 — **가장 효율적**
- **단점**: 안전 게이트·분업·리뷰 파이프라인 전무. 단일 대화 = 단일 에이전트.
- **우리와의 차이**: Aider는 **"빠르고 단순"**, 우리는 **"느리고 신중"**. 극단적 대비.
- **혼용 전략**: 소규모 수정은 Aider, 🔴 위험 파일 수정은 이 키트. 사용자가 판단해서 도구 전환.

### 4.5 Kiro / GitHub Spec Kit
- **규모**: Kiro는 IDE + CLI, Spec Kit은 CLI 템플릿
- **철학**: **spec-driven development** — `requirements.md` / `design.md` / `tasks.md`를 소스 오브 트루스로. EARS 표기법 사용.
- **장점**: 에이전트가 vague prompt 해석 안 해도 됨. 사양서가 계약서 역할.
- **단점**: 3개 파일 작성 자체가 부담. 기존 프로젝트(brownfield)에는 부적합.
- **우리와의 차이**: 우리 `/spec` 스킬이 **동일한 문제**를 풀지만, 별도 파일 대신 **대화 안에서 보고서** 로 해결. 가벼움 + 실용성.
- **혼용 전략**: Spec Kit으로 초기 spec 생성 → 우리 키트로 spec → plan → build 실행. 가능한 조합.

### 4.6 Claude Code 바닐라 + Anthropic Skills 카탈로그
- **규모**: Anthropic 공식 스킬 ~50개
- **철학**: 일반 목적 + 커뮤니티 스킬. `/debug`, `/simplify`, `/batch`, `/loop` 등 bundled.
- **장점**: 공식, 품질 보장, 업데이트 지속
- **단점**: 워크플로우가 없음. 개별 스킬은 있지만 "spec→ship 파이프라인"은 사용자가 구성.
- **우리와의 차이**: 우리는 **Anthropic 표준 위에 workflow 얹기**. 호환됨 — `/simplify`는 우리가 `/code-simplify`로 감싸서 같이 쓸 수 있음.

| **재사용성** | ★★★★☆ | 경로 변수화로 멀티 프로젝트 가능, 단 한국어 경로 가정 있음 |

## 8. 결론

**"안전하고 보수적인 한국 팀을 위한 체크리스트형 AI 파이프라인"** — 이 지점에서는 현재 시장에서 사실상 경쟁자가 없음. wshobson/BMAD가 대체할 수 없는 구멍을 차지.

다만 **단일 팀 수준을 넘어 확산하려면** 도메인 에이전트 추가, Windows 지원, 메트릭 수집, 테스트 커버리지 없이는 어려움. 현재는 **marketing-saas-v2의 1st party 도구**로 포지셔닝하고, 확산은 P1-P2 로드맵 완료 후 고려하는 게 현실적.
