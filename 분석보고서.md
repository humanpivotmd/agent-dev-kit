## 1. 디렉토리 구조 분석

**역할**: 재사용 가능한 Agent Development Kit의 정본(single source of truth). 모든 프로젝트는 여기서 복사해 사용.

### 📂 구성 (7개 파일)

#### 문서 (3개)
- **README.md** — 키트 소개, 설치법, 업그레이드 워크플로우. 원본은 `marketing-saas-v2`에서 2026-04-14 백업.
- **Agent-Development-Kit.md** — 5계층 아키텍처 문서: `CLAUDE.md → SKILLS → HOOKS → SUBAGENTS → PLUGINS`
- **Agent-Skills.md** — 6단계 워크플로우: `/spec → /plan → /build → /test → /review → /ship`. 각 단계의 입력/출력/체크리스트/전환 규칙 정의.

#### 실행 스크립트 (2개)
- **impact-analyzer.js** (358줄) — Node 기반 영향 분석기. 파일 의존성/UI/프로세스/함수/DB 5개 축을 grep 기반으로 스캔하고 🔴🟡🟢 리스크 점수 산출. `src/` 재귀 탐색해 usages 카운트. CLI: `node impact-analyzer.js <파일경로>`.
  - ⚠️ 미완성: `findExports`, `buildCallChain`, `findSchemas` 등 다수 메서드가 빈 배열 반환 스텁.
- **install.sh** — 새 프로젝트에 키트 설치. `impact-analyzer.js` 복사 + package.json에 axios 추가 + `.git/hooks/pre-commit` 생성(스테이징된 ts/tsx/js/jsx 최대 5개 자동 분석) + `.env.local` 템플릿.

#### 서브에이전트 정의 (`agents/`, 4개)
순차 파이프라인 — **설계 → 체크 → 구현 → 검증**. 각 단계는 역할 분리가 엄격하며 코드 수정 권한이 단일 에이전트(implementer)에만 있음.

| 에이전트 | 역할 | 금지 사항 |
|---------|------|---------|
| **designer.md** | 설계·위험도 분석·수정 순서 제안 | 코드/파일 생성 금지 |
| **checker.md** | grep으로 의존성 충돌·부작용 탐지 | 코드 수정 금지 |
| **implementer.md** | 실제 코드 작성, **3개 파일씩 배치** | 요청 외 파일 수정·라이브러리 설치 금지 |
| **verifier.md** | `npm run build` + `tsc --noEmit` + smoke test | 코드 직접 수정 금지 |

### 🔑 핵심 설계 원칙
1. **책임 분리**: 설계·체크·구현·검증이 각자의 에이전트로 분리되고, 각 단계 후 사용자 승인 게이트.
2. **3-파일 배치 규칙**: implementer는 3개 파일 단위로 중간 보고. 폭주 방지용 강제 체크포인트.
3. **🔴 위험 파일 재확인**: designer→checker→implementer가 위험 파일 포함 여부를 각 단계에서 재확인.
4. **사전 영향 분석**: pre-commit 훅으로 커밋 차단 가능한 hard gate.
5. **중앙 정본 모델**: 수정은 반드시 이 디렉토리에서 먼저, 각 프로젝트로 배포.

### ⚠️ 주의할 점
- `impact-analyzer.js`의 경로 가정이 `./src/` 고정 — Next.js `app/` 루트 구조에는 맞지 않을 수 있음.
- 다수 분석 메서드가 스텁이라 리스크 점수는 과소평가될 가능성.
- `install.sh`는 Bash 전용 — Windows에서는 Git Bash/WSL 필요.

## 3. 장점 (잘 수용한 부분)

### ✅ 1. 5계층 아키텍처 충실 반영
- `Agent-Development-Kit.md`가 **CLAUDE.md → SKILLS → HOOKS → SUBAGENTS → PLUGINS** 계층을 그대로 문서화 → 개념적 일관성 확보
- 각 계층의 책임 분리가 명확 (Pandey 모델의 "Layer별 독립성" 원칙 준수)

### ✅ 2. 4-에이전트 파이프라인의 실용적 단순화
- Pandey 모델은 subagent를 `code-reviewer / test-runner / explorer`로 예시만 제시 → 현재 키트는 **designer → checker → implementer → verifier**로 **구체화**해 바로 쓸 수 있음
- 각 에이전트의 "코드 수정 금지" 강제 → **"results only, delegate only"** 원칙(Layer 4) 실제 구현

### ✅ 3. Hooks를 하드 게이트로 실체화
- Pandey가 "PreToolUse/PostToolUse"로 추상화한 것을 **pre-commit hook + impact-analyzer.js**로 물리적으로 구현 → "Deterministic, Not AI" 원칙 준수
- Git hook = "Think Git hooks for your agent" 문구를 **문자 그대로** 실현

### ✅ 4. 3-파일 배치 규칙
- 참조 모델에는 없는 **독자적 안전장치**. "Small, atomic tasks"(모델 B) 원칙을 파일 개수 단위로 정량화 → 폭주 방지

### ✅ 5. 🔴🟡🟢 위험도 신호 체계
- 모델 B의 "/spec 단계 위험도 평가"를 3단계 컬러 코드로 단순화, 각 에이전트가 **재확인**하는 이중 체크 구조

## 4. 단점 (누락·약점)

### ❌ 1. PLUGINS 계층 부재 (Layer 5 미구현)
- 모델 A는 **"Marketplace (Team install) — npm 패키지로 스킬 배포"** 를 핵심으로 제시
- 현재 키트는 `cp` + `install.sh`로 수동 복사 → **버전 관리 없음, 롤백 불가, 팀 간 드리프트 필연**

### ❌ 2. SKILLS 계층이 문서화만 됨 (실행 불가)
- 모델 A: `SKILL.md`는 **description matching → auto-invoked → task-specific context** 로 자동 호출
- 현재: `Agent-Skills.md`는 **설명서**일 뿐 `/spec`, `/build` 같은 슬래시 커맨드가 **실제 존재하지 않음**
- 모델 B의 7개 커맨드 중 **`/code-simplify` 누락**

### ❌ 3. MCP Servers & Agent Teams 연동 전무
- 모델 A 양쪽에 그려진 **MCP(GitHub/DB/API) + Agent Teams(병렬 실행, 메시지 전달)** 전혀 없음
- 현재 파이프라인은 **순차적(sequential)** — designer→checker→implementer→verifier 직렬 진행 → **병렬 처리 기회 상실**

### ❌ 4. impact-analyzer.js의 구현 부채
- 모델 A의 "auto-lint on write, block rm -rf" 같은 **결정론적 가드**가 목표인데
- 실제 코드는 **스텁 투성이** (`findExports`, `buildCallChain`, `findSchemas` 등 빈 배열 리턴)
- `./src/` 경로 하드코딩 → Next.js `app/` 구조·모노레포에서 오작동
- grep 기반 문자열 매칭이라 **false positive 다발**

### ❌ 5. Context 계층 분리 부재 (Layer 2 원칙 위배)
- 모델 A: "SKILLS는 on-demand, 모듈화된 context chunks"
- 현재: `Agent-Skills.md`가 **7KB 단일 파일**에 모든 단계 설명 → 항상 전부 로드됨

### ❌ 6. 계층 간 피드백 루프 없음
- 현재 키트는 **단방향** — verifier가 실패하면 "implementer 재호출 안내"만 하고 사용자에게 떠넘김
- 자동 재시도·학습 루프 없음 → 매번 사용자 승인 병목

### ❌ 7. 정량 지표·모니터링 부재
- `Agent-Development-Kit.md`가 "성능 지표, 품질 지표"를 **언급만** 함 → 실제 측정 코드·대시보드 없음

### ❌ 8. 프로젝트 특화 결합 (재사용성 저하)
- `implementer.md`가 `src/lib/constants.ts`, `src/lib/auth-client.ts` 같은 **특정 프로젝트 구조**에 하드코딩
- 중앙 정본이라면서도 사실상 `marketing-saas-v2` 전용

### 2️⃣ SKILLS 자동 호출 불가 → **SKILL.md description matching 적용**

**해결책**: `Agent-Skills.md` 단일 7KB 파일을 6개 독립 SKILL.md로 분리하고 각각 description 기반 auto-invoke 설정.

```markdown
---
name: spec
description: Use when defining new feature requirements. Performs impact
  analysis across files, UI, processes, functions, DB, and APIs. Outputs
  risk-graded spec with 🔴🟡🟢 markers. Triggers before any /plan or /build.
---
```

**핵심 원칙**:
- Claude는 `available_skills`에서 description만 보고 호출 결정 → **description이 곧 트리거**
- "Complex, multi-step tasks reliably trigger skills" → 단순 쿼리는 트리거 안 됨
- description은 *언제 사용할지* 구체적으로 — "Explains X with Y" 형식 권장
- **on-demand 로딩**: 항상 로드되는 CLAUDE.md와 달리, 필요할 때만 컨텍스트 주입

**적용**: `/spec`, `/plan`, `/build`, `/test`, `/review`, `/code-simplify`, `/ship` → 각각 `skills/<name>/SKILL.md`로 분리

### 3️⃣ 병렬화 부재 → **Subagent 병렬 실행 패턴 도입**

**해결책**: verifier 단계를 **3개 병렬 subagent**로 쪼개기.

```
verifier (orchestrator)
 ├─ code-reviewer  (병렬)
 ├─ test-runner    (병렬)
 └─ security-scanner (병렬)
```

**구현 방식**:
- 명시적 호출: *"Check API routes, DB models, and frontend components simultaneously"*
- `@code-reviewer`, `@test-runner`는 Claude Code **빌트인 subagent** — 직접 만들 필요 없음
- 각 subagent에 **tool 권한 제한**: doc-reviewer는 Read+Grep만 → 안전성 보장
- HAMY 사례: **9개 병렬 reviewer**로 코드 리뷰 시간을 분→초로 단축

### 4️⃣ impact-analyzer.js 스텁 → **madge / ts-morph / Skott로 교체**

**해결책**: 자체 grep 구현을 폐기하고 검증된 AST 라이브러리 사용.

| 도구 | 용도 | 강점 |
|------|------|------|
| **madge** | 모듈 의존성 그래프, 순환 의존성 탐지, dead code | npm 표준, ts/tsx 지원 |
| **ts-morph** | TypeScript Compiler API 래퍼 | AST 직접 조작, **타입 정보 포함** 분석 |
| **Skott** | madge 후속, **7배 빠름** | 그래프 diffing, incremental 분석 |

**구체 교체안**:
```js
// 현재 (false positive 다발)
content.includes(fileName)

// 개선 (ts-morph)
const project = new Project();
const sourceFile = project.getSourceFile(targetFile);
const refs = sourceFile.getExportedDeclarations()
  .flatMap(d => d.findReferencesAsNodes());
```

**부가 효과**: ts-morph는 `tsconfig.json` 자동 인식 → 모노레포·`app/` 구조도 OK. 순환 의존성 자동 탐지 (madge built-in).

---

### 5️⃣ Hooks 결정론 강화 → **PreToolUse modify + PostToolUse validator**

**해결책**: 현재의 pre-commit hook(커밋 시점)을 **PreToolUse hook(파일 쓰기 시점)** 으로 앞당김.

**핵심 변화** (Claude Code v2.0.10+):
- **PreToolUse는 입력을 *수정* 가능** — 차단→재시도 루프 대신 JSON 입력을 고쳐서 통과
- PostToolUse는 결과 검증 후 피드백을 컨텍스트에 주입 → implementer가 즉시 학습

**hook event 매핑**:

| 시점 | 현재 키트 | 개선안 |
|------|---------|--------|
| 파일 쓰기 전 | ❌ 없음 | **PreToolUse**: 🔴 위험 파일 차단, 3-파일 규칙 강제 |
| 파일 쓰기 후 | ❌ 없음 | **PostToolUse**: lint/typecheck 즉시 실행, 결과 주입 |
| Subagent 종료 | ❌ 없음 | **SubagentStop**: designer→checker 자동 전환 |
| 커밋 시점 | ✅ pre-commit | 유지 |

> *"Hooks guarantee behavior; prompts suggest it."*

---

### 6️⃣ MCP Servers 연동 → **GitHub + DB MCP 표준 채택**

**즉시 적용 가능한 MCP 서버**:
1. **`github/github-mcp-server`** (공식) — designer가 관련 이슈/PR 조회, checker가 과거 변경 이력 추적
2. **PostgreSQL/SQLite MCP** — DB 스키마 변경 영향 분석 시 read-only 쿼리로 실제 스키마 조회
3. **`zilliztech/claude-context`** — 코드베이스 시맨틱 검색 → grep보다 정확한 영향 분석

**연결 예시**:
```json
// .claude/mcp.json
{
  "mcpServers": {
    "github": { "command": "npx", "args": ["@github/mcp-server"] },
    "db": { "command": "npx", "args": ["@modelcontextprotocol/server-postgres", "$READONLY_DB_URL"] }
  }
}
```

### 7️⃣ 피드백 루프 부재 → **PostToolUse + SubagentStop 자동 재시도**

```bash
# .claude/hooks/post-tool-use.sh
if ! npx tsc --noEmit; then
  echo '{"continue": true, "feedback": "Type errors detected, fix and retry"}'
fi
```

PostToolUse는 **`tool_input`과 `tool_response`를 모두 받음** → 어떤 파일이 어떤 에러를 냈는지 정확히 피드백 가능.

| **P2** | GitHub MCP + DB MCP 연결 | designer 컨텍스트 풍부화 | 하 |
| **P3** | implementer 경로 변수화 | 재사용성 | 하 |

**핵심 인사이트**: 8개 약점 중 **6개는 Claude Code가 이미 제공하는 표준 기능**(skills, hooks, subagents, plugins, MCP)을 안 쓰고 자체 구현한 데서 발생. 자체 구현을 표준으로 교체하면 코드량은 줄고 기능은 늘어남.
