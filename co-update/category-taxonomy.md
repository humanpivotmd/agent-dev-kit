# Co-update Category Taxonomy — 표준 분류 체계

> **목적**: 사례 카테고리를 표준화해서 프로젝트 간 비교·이전·집계 가능하게.
> **사용**: `cases.md`의 `**카테고리**:` 필드에 이 표의 태그 사용.
> **확장**: 새 카테고리 추가 가능. 단 기존과 겹치지 않게.

---

## 11개 표준 카테고리

### 1. `auth/role`
**의미**: 인증, 권한, role/permission 관련 변경.
**예시**:
- 새 사용자 role 추가
- 권한 체크 누락
- RLS 정책 미반영
- 비밀번호 변경 흐름

### 2. `db-schema`
**의미**: 테이블, 컬럼, 제약, 마이그레이션 등 데이터베이스 구조 변경.
**예시**:
- 새 컬럼 추가했지만 types.ts 누락
- CHECK constraint 변경했지만 코드 미반영
- 마이그레이션과 코드 정합성 어긋남

### 3. `data-integrity`
**의미**: 런타임 데이터 정합성 — 중복, race condition, NULL 처리.
**예시**:
- 같은 row 중복 INSERT
- React key 충돌 (중복 데이터)
- NULL을 string으로 가정해서 깨짐

### 4. `ui-flow`
**의미**: 네비게이션, 진행 바, 페이지 순서, 라우팅.
**예시**:
- 진행 바에 새 step 누락
- 페이지 추가했지만 사이드바 메뉴 누락
- 라우팅 가드 미적용

### 5. `form`
**의미**: 입력 필드, 폼 검증, 제출 처리.
**예시**:
- Zod 스키마 누락
- sanitize 미적용
- 토스트/로딩 누락
- 접근성 (aria) 누락

### 6. `entry-point`
**의미**: 새 버튼·링크가 새로운 데이터 흐름을 시작하는 경우.
**예시**:
- "만들기" 버튼 추가했지만 settings_snapshot 누락
- 진입점 마이그레이션 시 옵션 누락
- 데드 진입점 (어디서도 링크 안 됨)

### 7. `pipeline-step`
**의미**: 다중 단계 워크플로우의 단계 추가/제거/순서 변경.
**예시**:
- STEP 빼라고 했는데 step_status는 그대로
- 새 STEP 추가했지만 진행 바·가드·API 매핑 누락
- 단계 진입 게이트가 너무 엄격해서 부분 사용 시나리오 막힘

### 8. `env-config`
**의미**: 환경 변수, 배포 설정, 빌드 설정.
**예시**:
- env 변수 추가했지만 .env.example 누락
- next.config.ts 빌드 모드 vs 실행 모드 불일치
- Railway/Vercel 환경에 미반영

### 9. `legacy-debt`
**의미**: 설계 문서와 코드의 불일치, 오래된 코드 경로.
**예시**:
- 설계엔 STEP6 있지만 실제 코드는 빈 redirect
- 옛 페이지가 데드 코드로 남음
- 두 진입점이 다른 동작

### 10. `support-burden`
**의미**: 운영 부담 — admin 도구, 감사 로그, 수동 개입.
**예시**:
- admin 액션 추가했지만 audit log 누락
- 사용자 stuck state 발생 시 admin이 SQL 직접 실행해야 함
- support ticket 처리 도구 부재

### 11. `dead-code`
**의미**: 어디서도 참조되지 않는 코드, 호출 안 되는 함수, import 안 되는 파일.
**예시**:
- 옛 페이지가 라우팅에서 분리됐지만 파일 남음
- 사용 안 되는 export
- 빈 redirect만 있는 페이지

## 카테고리 사용 규칙

### 1. **다중 태그 가능**
한 사례는 보통 2~3개 카테고리 매칭. 콤마로 구분:
```markdown
- **카테고리**: `entry-point`, `ui-flow`, `pipeline-step`
```

### 2. **새 카테고리는 신중히**
기존 11개로 안 되는 경우만 추가. 추가 시:
- 이 파일에 정의 + 예시
- 다른 프로젝트도 같이 사용할 수 있게 generic하게

### 3. **빈도 임계값**
한 카테고리에 **3건+ 누적**되면 `pattern-extractor.mjs`가 패턴 보강 제안.

### 4. **프로젝트 간 비교**
같은 카테고리의 사례를 여러 프로젝트에서 모으면 **공통 패턴 발견** 가능.
미래의 학습 시스템에서 활용.

## 마이그레이션 (기존 사례 → 표준 카테고리)

`cases.md`의 기존 카테고리를 이 표준으로 변환할 때:

| 기존 한국어 태그 | 표준 영어 태그 |
|---|---|
| 진입점추가, 진입점이전 | `entry-point` |
| 파이프라인단계, 진입게이트 | `pipeline-step` |
| 진행바 | `ui-flow` |
| 데이터정합성, 중복row | `data-integrity` |
| React키 | `data-integrity` |
| 사용자시나리오, 풀세트가정 | `pipeline-step` |
| 데드코드 | `dead-code` |
| 기능손실 | `entry-point`, `legacy-debt` |
| 설계갭 | `legacy-debt` |
| legacy, 미해결 | `legacy-debt` |

이 매핑은 **참고용**. 실제로는 사례 내용을 보고 가장 맞는 태그 선택.
